# 架构总览（单机版）

## 设计目标

基于 Virtual Actor 模型的事件溯源系统，单机部署：
- WASM 组件承载领域逻辑（参数校验[可选]、命令处理[必须]、事件应用[必须]）
- Host Server 负责基础设施（GraphQL、Actor 生命周期、事件存储）
- 领域事件作为唯一事实来源，遵循领域事件最佳实践
- **强一致性**：命令响应前事件必须持久化，零数据丢失
- **Virtual Actor**：聚合按需激活/自动休眠，内存受限下仅热点聚合常驻

## 一致性承诺

**铁律：响应客户端之前，事件必须已持久化到 Event Store。**

```
客户端收到成功响应 → 事件一定已落盘
客户端收到失败响应 → 无副作用（状态未变）
客户端超时（未收到响应） → 重新查询聚合状态确认命令是否已生效
```

## 整体架构

```
┌───────────────────────────────────────────────────────┐
│              GraphQL API (axum)                         │
│        Query (查询)    │    Mutation (命令入口)         │
└────────────┬───────────┴──────────────┬───────────────┘
             │                          │
             │             ┌────────────▼──────────────┐
             │             │  Virtual Actor Runtime     │
             │             │                            │
             │             │  ┌─ active ──────────────┐│
             │             │  │ agg-001 → Actor (hot)  ││
             │             │  │ agg-002 → Actor (warm) ││
             │             │  └────────────────────────┘│
             │             │                            │
             │             │  内存预算: max_active 聚合  │
             │             │  激活: 快照+事件 → 内存     │
             │             │  休眠: 内存 → 快照 → 释放   │
             │             └────────────┬──────────────┘
             │                          │ 同步写入
    ┌────────┴──────────────────────────▼──────────────┐
    │        PostgreSQL (事件 + 快照 + 幂等键)           │
    └──────────────────────────────────────────────────┘
```

## 核心设计原则

| 原则 | 实现方式 |
|------|----------|
| 强一致性 | 事件持久化到 DB 确认后才响应客户端 |
| 按需激活 | 首次访问时从快照+事件恢复，后续命令直接操作内存状态 |
| 自动休眠 | 空闲超时或内存压力时保存快照并释放内存 |
| 位置透明 | 调用方只需 aggregate_id，运行时自动激活/路由 |
| 无锁串行 | 每个 Virtual Actor 单线程处理，天然无并发 |
| 故障恢复 | Actor 崩溃后从快照+事件自动重新激活 |

## 职责划分

| 层 | 职责 | 技术选型 |
|----|------|----------|
| GraphQL | 接收命令、返回结果 | async-graphql + axum |
| Virtual Actor Runtime | 聚合激活/休眠/寻址/内存管理 | Rust (tokio mpsc channel 单线程循环) |
| Virtual Actor | 单聚合状态管理、同步持久化 | tokio mpsc channel |
| WASM 组件 | 参数校验、命令决策、事件应用 | Wasmtime（每次调用创建实例） |
| Event Store | 事件追加 + 快照 + 幂等键 | PostgreSQL |

## 查询路径说明

本设计仅覆盖命令（写入）路径。聚合状态的查询/读取由独立的查询模块承担，
不在本设计范围内。查询模块可通过以下方式实现：
- 直接查询 Event Store 重建状态
- 订阅事件流构建读模型（CQRS 投影）
- 通过 VirtualActorRuntime 获取活跃聚合的内存状态

## 性能目标（单机 PostgreSQL）

| 指标 | 目标值 | 前提条件 |
|------|--------|----------|
| 吞吐 | ~10,000 TPS | 1 PG 实例，无实例池 |
| 热聚合命令延迟 P99 | < 20ms | 本地 PG，SSD |
| 冷聚合激活延迟 P99 | < 50ms | 快照存在 |

注：无 WASM 实例池时每次调用有 ~100-500μs 实例化开销，吞吐低于完整版。
后续可通过引入实例池提升至 ~30,000 TPS。

## 内存预算模型

```
Actor 可用内存 = 总可用内存 - WASM 运行时基础开销
max_active = Actor 可用内存 / (平均聚合状态大小 + Actor 开销 ~1KB)
```

| 可用内存 | 聚合状态 | max_active |
|----------|---------|-----------|
| 4 GB | 1 KB | ~200万 |
| 16 GB | 1 KB | ~800万 |
| 16 GB | 10 KB | ~80万 |
