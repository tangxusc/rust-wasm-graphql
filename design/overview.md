# 架构总览

## 设计目标

在现有 WASM-GraphQL 自动映射架构基础上，引入 DDD 事件溯源模式：
- WASM 组件承载领域逻辑（参数校验[可选]、状态重建[可选]、命令处理[必须]）
- Host Server 负责基础设施（存储、发布、Virtual Actor 生命周期管理）
- 领域事件作为唯一事实来源
- **强一致性保证**：命令响应前事件必须持久化，零数据丢失
- **Virtual Actor 模型**：聚合按需激活/自动休眠，内存受限下仅热点聚合常驻
- **位置透明寻址**：调用方无需关心聚合是否在内存中，运行时自动管理
- **同步持久化**：事件写入 DB 确认后才响应客户端

## 一致性承诺

**命令处理的铁律：响应客户端之前，事件必须已持久化到 Event Store。**

```
客户端收到成功响应 → 事件一定已落盘
客户端收到失败响应 → 无副作用（状态未变）
客户端超时（未收到响应） → 通过幂等键重试安全确认
```

绝不出现"客户端认为成功但数据丢失"的情况。

## 整体架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          GraphQL API (axum)                               │
│              Query (查询)           │    Mutation (命令入口)               │
└──────────────────┬─────────────────┴──────────────────┬─────────────────┘
                   │                                     │
                   │                        ┌───────────▼──────────────┐
                   │                        │  Command Gateway         │
                   │                        │  幂等检查(布隆+KV)       │
                   │                        └───────────┬──────────────┘
                   │                                     │
                   │                        ┌────────────▼─────────────────┐
                   │                        │  Virtual Actor Runtime        │
                   │                        │                               │
                   │                        │  ┌─ active ─────────────────┐│
                   │                        │  │ agg-001 → Actor (hot)    ││
                   │                        │  │ agg-002 → Actor (hot)    ││
                   │                        │  │ agg-003 → Actor (warm)   ││
                   │                        │  └──────────────────────────┘│
                   │                        │                               │
                   │                        │  内存预算: max_active 聚合    │
                   │                        │  激活: 快照+事件 → 内存       │
                   │                        │  休眠: 内存 → 快照 → 释放     │
                   │                        │                               │
                   │                        │  ┌─ WASM Instance Pool ─────┐│
                   │                        │  │ validate-X(可选) /       ││
                   │                        │  │ handle-X(必须) /         ││
                   │                        │  │ apply-events(可选)       ││
                   │                        │  └──────────────────────────┘│
                   │                        └────────────┬─────────────────┘
                   │                                     │ 同步写入（等待确认）
          ┌────────┴─────────────────────────────────────▼─────────────────┐
          │           Event Store (当前：单实例 PG，未来：分片集群)            │
          └────────────────────────────────────┬───────────────────────────┘
                                               │ CDC (WAL 监听)
                                      ┌────────▼────────┐
                                      │     Kafka       │
                                      └─────────────────┘
```

## Virtual Actor vs 传统 Actor

| 维度 | 传统 Actor | Virtual Actor（当前方案） |
|------|-----------|--------------------------|
| 生命周期 | 常驻内存，手动管理 | 按需激活，自动休眠 |
| 内存占用 | 所有聚合 × 状态大小 | 仅活跃聚合 × 状态大小 |
| 寻址方式 | hash 分片到固定 Actor | 通过 aggregate_id 透明寻址 |
| 粒度 | 一个 Actor 管理一个分片 | 一个聚合 = 一个 Virtual Actor |
| 扩缩容 | 需要 rehash | 运行时自动重新分布 |
| 内存受限 | 内存爆炸 | 内存预算 + LRU 驱逐 |

## 核心设计原则

| 原则 | 实现方式 |
|------|----------|
| 强一致性 | 事件持久化到 DB 确认后才响应客户端，零数据丢失 |
| 按需激活 | 首次访问时从快照+事件恢复，后续命令直接操作内存状态 |
| 自动休眠 | 空闲超时或内存压力时保存快照并释放内存 |
| 位置透明 | 调用方只需 aggregate_id，运行时自动激活/路由 |
| 无锁串行 | 每个 Virtual Actor 单线程处理，天然无并发 |
| 背压控制 | Actor 邮箱有界，满时拒绝新请求（503） |
| 故障恢复 | Actor 崩溃后从快照+事件自动重新激活，无数据丢失 |
| 内存预算 | max_active 限制活跃聚合数量，LRU 驱逐冷聚合 |

## 职责划分

| 层 | 职责 | 技术选型 |
|----|------|----------|
| GraphQL | 接收命令、返回结果、提供查询 | async-graphql + axum |
| Command Gateway | 幂等检查、背压控制 | Rust (host-server) |
| Virtual Actor Runtime | 聚合激活/休眠/寻址/内存管理 | Rust (DashMap + tokio spawn) |
| Virtual Actor | 单聚合状态管理、同步持久化 | tokio mpsc channel |
| WASM Instance Pool | 参数校验、命令决策、事件应用 | Wasmtime 预热实例池 |
| Event Store | 事件追加、同步写入 | PostgreSQL（当前单实例，未来可分片） |
| Snapshot Store | 快照点查点写、Actor 恢复/休眠 | KV 存储 (Redis / sled) |
| Event Publisher | CDC 捕获 + Kafka 发布 | Debezium / WAL 监听 |

## 性能目标

### 当前阶段（单实例 PostgreSQL）

| 指标 | 目标值 | 前提条件 |
|------|--------|----------|
| 单机吞吐 | ~30,000 TPS | 1 PG 实例 |
| 热聚合命令延迟 P99 | < 15ms（内存命中 + 同步写 DB） | 本地 PG，SSD |
| 冷聚合激活延迟 P99 | < 50ms（快照恢复） | 快照存在 |
| Actor 激活时间 | < 10ms（有快照）/ < 500ms（无快照） | — |

### 未来演进（分片 + 集群）

| 指标 | 目标值 | 前提条件 |
|------|--------|----------|
| 单机吞吐（多分片） | 60,000 - 120,000 TPS | 2-4 本地 PG 实例 |
| 集群吞吐 | 500,000+ TPS | 16+ 分片，多节点 |

注：
- 同步写入延迟（PG commit + fsync）通常 1-5ms，加上 WASM 调用和 Actor 调度，P99 < 15ms 是务实目标
- 未来通过 Event Store 分片 + Virtual Actor 并行弥补同步写入的吞吐开销
- 集群方案详见 [cluster.md](./cluster.md)

## 内存预算模型

```
总可用内存 = WASM 实例池内存 + Actor 内存
WASM 实例池内存 = 模块数 × 池大小 × 单实例内存（含线性内存，通常 1-4 MB）
Actor 可用内存 = 总可用内存 - WASM 实例池内存
max_active = Actor 可用内存 / (平均聚合状态大小 + Actor 开销 ~1KB)
```

| 可用内存 | WASM 池占用 | Actor 可用 | 聚合状态 | max_active | 说明 |
|----------|------------|-----------|---------|-----------|------|
| 4 GB | ~256 MB (4模块×16实例×4MB) | ~3.7 GB | 1 KB | ~185万 | 小型部署 |
| 16 GB | ~512 MB (4模块×32实例×4MB) | ~15.5 GB | 1 KB | ~775万 | 中型部署 |
| 16 GB | ~512 MB | ~15.5 GB | 10 KB | ~77万 | 大状态聚合 |
| 64 GB | ~1 GB (8模块×32实例×4MB) | ~63 GB | 1 KB | ~3150万 | 大型部署 |

注：WASM 实例的线性内存大小取决于组件复杂度，简单组件可能仅 64KB-256KB，
复杂组件（含大量静态数据）可达 4MB+。部署前应实测单实例内存占用。

## 查询侧设计（CQRS 读模型）

本系统采用 CQRS 模式，命令侧（Mutation）和查询侧（Query）分离：

### 查询模块（WASM 直接查询）

非聚合的 WASM 组件可导出普通函数，Host 自动映射为 GraphQL Query 字段。
这类查询适用于无状态计算或简单数据转换，不涉及事件溯源。

```wit
package example:utils;

interface queries {
    calculate-tax: func(amount: u64, rate: f32) -> u64;
    format-address: func(parts: list<string>) -> string;
}

world utils-queries {
    export queries;
}
```

### 读模型投影（事件驱动）

聚合的查询通过事件消费者构建读模型，与命令侧完全解耦：

```
命令侧：GraphQL Mutation → Actor → Event Store
                                        │ CDC
                                        ▼
查询侧：Kafka → 投影消费者 → 读模型 DB → GraphQL Query
```

读模型的构建和查询不在本系统核心范围内，由下游消费者自行实现。
Host Server 仅负责：
1. 命令处理（Mutation）— 通过 Virtual Actor + Event Store
2. 无状态查询（Query）— 通过普通 WASM 组件直接调用
3. 事件发布（CDC → Kafka）— 供下游消费者构建读模型

### 为什么不在 Host 内建读模型

| 考量 | 说明 |
|------|------|
| 职责单一 | Host 专注命令处理和事件持久化，读模型由专用服务承担 |
| 独立扩缩 | 读写负载特征不同，分离后可独立扩缩容 |
| 技术选型自由 | 读模型可用 Elasticsearch、ClickHouse 等最适合查询模式的存储 |
| 最终一致性 | 读模型天然接受最终一致，无需与命令侧共享强一致保证 |

> 注：查询侧投影（Projection）不在本系统核心范围内，相关设计说明已合并至本节。
> 下游消费者可参考 [event-publishing.md](./event-publishing.md) 中的消费者端设计获取事件流接入方式。
