# 架构总览

## 设计目标

在现有 WASM-GraphQL 自动映射架构基础上，引入 DDD 事件溯源模式：
- WASM 组件承载全部领域逻辑（参数校验、状态重建、命令处理）
- Host Server 负责基础设施（存储、发布、Virtual Actor 生命周期管理）
- 领域事件作为唯一事实来源
- **强一致性保证**：命令响应前事件必须持久化，零数据丢失
- **Virtual Actor 模型**：聚合按需激活/自动休眠，内存受限下仅热点聚合常驻
- **位置透明寻址**：调用方无需关心聚合是否在内存中，运行时自动管理
- **同步持久化**：事件写入 DB 确认后才响应客户端

## 一致性承诺

**命令处理的铁律：响应客户端之前，事件必须已持久化到 Event Store。**

```
客户端收到成功响应 → 事件一定已落盘
客户端收到失败响应 → 无副作用（状态未变）
客户端超时（未收到响应） → 通过幂等键重试安全确认
```

绝不出现"客户端认为成功但数据丢失"的情况。

## 整体架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          GraphQL API (axum)                               │
│              Query (读模型/投影)    │    Mutation (命令入口)               │
└──────────────────┬─────────────────┴──────────────────┬─────────────────┘
                   │                                     │
          ┌────────▼────────┐               ┌───────────▼──────────────┐
          │  Projection     │               │  Command Gateway         │
          │  (读取投影视图)  │               │  幂等检查(布隆+KV)       │
          └────────┬────────┘               └───────────┬──────────────┘
                   │                                     │
                   │                        ┌────────────▼─────────────────┐
                   │                        │  Virtual Actor Runtime        │
                   │                        │                               │
                   │                        │  ┌─ active ─────────────────┐│
                   │                        │  │ agg-001 → Actor (hot)    ││
                   │                        │  │ agg-002 → Actor (hot)    ││
                   │                        │  │ agg-003 → Actor (warm)   ││
                   │                        │  └──────────────────────────┘│
                   │                        │                               │
                   │                        │  内存预算: max_active 聚合    │
                   │                        │  激活: 快照+事件 → 内存       │
                   │                        │  休眠: 内存 → 快照 → 释放     │
                   │                        │                               │
                   │                        │  ┌─ WASM Instance Pool ─────┐│
                   │                        │  │ validate / handle /      ││
                   │                        │  │ apply-events             ││
                   │                        │  └──────────────────────────┘│
                   │                        └────────────┬─────────────────┘
                   │                                     │ 同步写入（等待确认）
          ┌────────┴─────────────────────────────────────▼─────────────────┐
          │                    Event Store (分片)                            │
          └────────────────────────────────────┬───────────────────────────┘
                                               │ CDC (WAL 监听)
                                      ┌────────▼────────┐
                                      │     Kafka       │
                                      └─────────────────┘
```

## Virtual Actor vs 传统 Actor

| 维度 | 传统 Actor | Virtual Actor（当前方案） |
|------|-----------|--------------------------|
| 生命周期 | 常驻内存，手动管理 | 按需激活，自动休眠 |
| 内存占用 | 所有聚合 × 状态大小 | 仅活跃聚合 × 状态大小 |
| 寻址方式 | hash 分片到固定 Actor | 通过 aggregate_id 透明寻址 |
| 粒度 | 一个 Actor 管理一个分片 | 一个聚合 = 一个 Virtual Actor |
| 扩缩容 | 需要 rehash | 运行时自动重新分布 |
| 内存受限 | 内存爆炸 | 内存预算 + LRU 驱逐 |

## 核心设计原则

| 原则 | 实现方式 |
|------|----------|
| 强一致性 | 事件持久化到 DB 确认后才响应客户端，零数据丢失 |
| 按需激活 | 首次访问时从快照+事件恢复，后续命令直接操作内存状态 |
| 自动休眠 | 空闲超时或内存压力时保存快照并释放内存 |
| 位置透明 | 调用方只需 aggregate_id，运行时自动激活/路由 |
| 无锁串行 | 每个 Virtual Actor 单线程处理，天然无并发 |
| 背压控制 | Actor 邮箱有界，满时拒绝新请求（503） |
| 故障恢复 | Actor 崩溃后从快照+事件自动重新激活，无数据丢失 |
| 内存预算 | max_active 限制活跃聚合数量，LRU 驱逐冷聚合 |

## 职责划分

| 层 | 职责 | 技术选型 |
|----|------|----------|
| GraphQL | 接收命令、返回结果、提供查询 | async-graphql + axum |
| Command Gateway | 幂等检查、背压控制 | Rust (host-server) |
| Virtual Actor Runtime | 聚合激活/休眠/寻址/内存管理 | Rust (DashMap + tokio spawn) |
| Virtual Actor | 单聚合状态管理、同步持久化 | tokio mpsc channel |
| WASM Instance Pool | 参数校验、命令决策、事件应用 | Wasmtime 预热实例池 |
| Event Store | 分片事件追加、同步写入 | PostgreSQL 分片集群 |
| Snapshot Store | 快照点查点写、Actor 恢复/休眠 | KV 存储 (Redis / sled) |
| Event Publisher | CDC 捕获 + Kafka 发布 | Debezium / WAL 监听 |
| Projection | 消费事件构建读模型 | 独立消费者进程 |

## 性能目标

| 指标 | 目标值 |
|------|--------|
| 单机吞吐（热聚合） | 50,000 - 100,000 TPS |
| 集群吞吐 | 500,000+ TPS |
| 热聚合命令延迟 P99 | < 10ms（内存命中 + 同步写 DB） |
| 冷聚合激活延迟 P99 | < 50ms（快照恢复） |
| Actor 激活时间 | < 10ms（有快照）/ < 500ms（无快照） |

注：通过 Event Store 分片 + Virtual Actor 并行弥补同步写入的吞吐开销。

## 内存预算模型

```
max_active = 可用内存 / (平均聚合状态大小 + Actor 开销 ~1KB)
```

| 可用内存 | 聚合状态 | max_active | 说明 |
|----------|---------|-----------|------|
| 4 GB | 1 KB | ~200万 | 小型部署 |
| 16 GB | 1 KB | ~800万 | 中型部署 |
| 16 GB | 10 KB | ~80万 | 大状态聚合 |
| 64 GB | 1 KB | ~3200万 | 大型部署 |
